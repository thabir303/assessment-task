## 2026-07-18 — Real VM stop/resume and retryable-failure implementation

- **Session goal:** Fix the top-priority repair identified in the prior verification session: implement real, adversarially-tested (not happy-path-only) VM stop/resume and retryable-failure recovery. `tool-websearch` was explicitly left unimplemented per instruction (suggestions only); `docs/DEMO_SCRIPT.md` was rewritten to cover the full assessment for an upcoming cap.so recording.
- **Implementation summary:**
  - `convex/lib/daytona.ts`: extracted `createSandboxAndStartRunner` (shared by first-time provisioning and the resume fallback path) and `startRunnerProcess` (defensively checks `process.listSessions()` before `createSession`, since Daytona's in-sandbox session registry is undocumented runtime state, not guaranteed to survive a stop/start). Sandbox creation no longer sets `ephemeral: true` (which forced `autoDeleteInterval: 0` — "delete immediately upon stopping", making stop/resume impossible by construction); `autoDeleteInterval: 1440` (24h) is set explicitly instead, as a bounded cost safety net.
  - `convex/actions/provisionSandbox.ts`: refactored to call the shared helper (no behavior change beyond the ephemeral fix).
  - `convex/actions/stopSandbox.ts` (new): calls `sandbox.stop(60)` on the thread's recorded sandbox; falls back to `markThreadError` if the sandbox is already gone.
  - `convex/actions/resumeSandbox.ts` (new): checks whether the thread's original sandbox still exists and is `stopped`/`started`; if so, restarts the runner in it and re-fetches a live preview link (never trusts the pre-stop link/token, since the SDK gives no guarantee they survive a restart) before marking the thread `ready`. If the sandbox is genuinely gone, falls back to `createSandboxAndStartRunner` (a fresh sandbox, fresh Pi session — the thread's Convex message history is untouched either way).
  - `convex/threads.ts`: added `stop` (`ready -> stopped`) and `resume` (`stopped|error -> provisioning`, both already-valid transitions in the existing `allowedThreadTransitions` map — no lifecycle/schema changes were needed) mutations, each scheduling the corresponding action.
  - `sandbox/runner/src/server.ts`: **the actual session-persistence fix.** `SessionManager.create(cwd)` never resumes — confirmed by reading `@mariozechner/pi-coding-agent`'s source — it always starts a brand-new empty session even when a prior session file for that `cwd` exists on disk. Changed to `SessionManager.continueRecent(cwd)`, which loads the most recent session file for that `cwd` if one exists and transparently falls back to a fresh session when there isn't one (verified this fallback in the library source before relying on it, so first-time provisioning of a brand-new sandbox is unaffected).
  - `apps/web/src/app/page.tsx` / `globals.css`: added **Stop VM** / **Resume VM** / **Retry** controls in the chat header, shown based on `thread.state`.
  - **A second real bug found and fixed via adversarial testing, not code review:** `convex/actions/consumeRunnerStream.ts` only treated a *thrown* fetch/read error as a transport failure. Killing a real sandbox precisely mid-stream (via a poll-loop that watched `messages:list` for the first partial assistant chunk before deleting the sandbox, rather than guessing a sleep duration) revealed a case where the HTTP stream instead ends cleanly (`done: true`) without ever emitting a `run_terminal` event — and that was silently letting the thread revert to `ready` with a dead `sandboxId`, exactly the failure mode the *original* bug fix (recorded in the prior session) was supposed to prevent, just via a different code path. Fixed by tracking `receivedTerminalEvent` and treating its absence as a transport failure regardless of whether the stream threw or just quietly ended.
  - The Daytona snapshot (`pi-agent-v1`) was rebuilt via `npm run provision:snapshot` so the `continueRecent` runner fix is actually baked into new sandboxes (the snapshot is prebuilt; source changes to the runner do nothing until it's rebuilt).
